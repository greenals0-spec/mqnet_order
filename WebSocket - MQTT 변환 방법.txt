[목표] 현재 FastAPI WebSocket 기반으로 동작하는 실시간 통신망을, 향후 100개 이상 매장의 트래픽 처리를 위해 MQTT 아키텍처로 전면 개편하고자 해. 과거에 통신 방식을 바꾸면서 시스템 전체가 마비되는 심각한 오류를 겪었기 때문에, 이번에는 극도로 안전하고 단계적인(Phased) 접근을 해야 해.

[작업 원칙 및 지시사항] 1. 기존 코드 파괴 금지 (병행 운용) 처음부터 기존 WebSocket 코드를 지우지 마. 먼저 MQTT 브로커(Mosquitto) 환경을 세팅하고, 기존 로직은 둔 채로 MQTT 코드를 '추가'만 해서 두 시스템이 동시에 돌아가도록 만들어줘. 완벽히 테스트가 끝난 후에 기존 웹소켓을 걷어낼 거야.

2. 페이로드(데이터) 구조 통일 대기, 호출, 주차, 주문 내역을 개별적인 API나 소켓으로 나누지 말고, topic과 message payload 구조를 하나로 통일해 줘. (예: store/{store_id}/call, store/{store_id}/order) 프론트엔드에서는 하나의 MQTT 클라이언트만 유지하도록 해.

3. 촘촘한 체크포인트(Checkpoint) 로깅 필수 ⭐ 데이터가 이동하는 모든 병목에 반드시 명시적인 콘솔 로그를 남겨줘.

모바일에서 Publish 할 때 [MQTT 송신]
백엔드에서 Subscribe로 받을 때 [MQTT 수신 - 백엔드]
DB에 저장 성공/실패 시 [DB 저장 상태]
백엔드에서 카운터/주방으로 Publish 할 때 [MQTT 브로드캐스트]
카운터/주방 프론트엔드에서 데이터 받을 때 [MQTT 수신 - 카운터] 에러가 나면 화면이 멈추게 하지 말고, 어떤 체크포인트에서 죽었는지 로그에 명확히 찍히게 예외 처리를 철저히 해.
4. 단일 기능부터 마이그레이션 (기능 쪼개기) 모든 기능을 한 번에 바꾸지 마. 제일 먼저 '직원 호출(Staff Call)' 기능 딱 하나만 MQTT로 통신하도록 구현해 줘. 이게 완벽하게 작동하고 체크포인트 로그가 정상적으로 찍히는 걸 내 눈으로 확인하면, 그때 다음 기능(주차, 대기, 주문)으로 넘어갈 거야. 알겠지?
