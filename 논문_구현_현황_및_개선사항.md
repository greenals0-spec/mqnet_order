## 1. CodeLess 논리 매트릭스 ⚠️ 30%

논문 정의: [노출, 입력, 버튼, 분기점] 4요소를 매트릭스로 정의하여 코드 수정 없이 운영 로직을 변경

### ✅ 구현된 부분
- `LogicInventory.tsx` — 5단계 수명 주기 매트릭스(STEP 0~4) 정적 시각화 존재
- `StoreManualEditor.tsx` — 운영자가 매장 규칙을 자유 텍스트로 작성 → AI 해석 (`/api/store/manual`)
- 버튼 중심 상태 머신: 주방 완료 버튼, 서비스 완료 버튼, 정산 완료 버튼으로 상태 전이

### ⚠️ 부분 구현
- StoreManualEditor가 서술형 텍스트만 지원 → AI가 해석하나 프로그래밍적 강제 실행은 안 됨
- 역할(Role)별 로직이 컴포넌트 코드에 하드코딩 (변경 시 재배포 필요)

### ❌ 미구현 / 개선 필요
- [ ] **비주얼 룰 에디터 없음** — IF/THEN 조건 빌더 UI (드롭다운 필드 + 연산자 + 액션)
- [ ] **분기점 동적 설정 불가** — 선결제/후결제, 포장/매장 조건 등이 코드에 고착됨
- [ ] **룰 버전 관리 없음** — 규칙 변경 이력, A/B 테스트, 롤백 불가
- [ ] **룰 테스트 모드 없음** — 배포 전 시뮬레이션 검증 불가

## 2. AI Knowledge Pool ✅ 80%

논문 정의: AI가 상황 데이터를 실시간으로 해석·저장하는 'No-DB 지향' 유연한 지식 저장소

### ✅ 구현된 부분
- **Bundle 구조** (`types.ts`) — 12가지 타입 (Orders, Menus, Waiting, Checkins, Reservations, Settlement 등)
- **핵심 필드** — `type`, `status`, `items[]`, `timestamp`, `store_id`, `table_id`, `session_id`
- **`/api/pool`** 엔드포인트로 전체 번들 중앙 조회 (`useSituation` 훅)
- **MQTT 실시간 스트리밍** — `store/{store_id}/kitchen` 토픽으로 번들 실시간 갱신
- **AI 처리 워크플로우** — 텍스트 입력 → `POST /api/situation` → AI 해석 → 번들 자동 생성
- `AdminDashboard.tsx` — 번들 집계 기반 매출/주문/직원 KPI 표시

### ⚠️ 부분 구현
- 지식 풀 필터링이 클라이언트 사이드에서 처리 (서버 측 시맨틱 쿼리 없음)
- AI 해석 과정이 백엔드에서만 진행 → 프론트에서 신뢰도/근거 확인 불가

### ❌ 미구현 / 개선 필요
- [ ] **지식 풀 전용 뷰 없음** — 번들을 의미적으로 탐색·검색하는 UI 부재
- [ ] **시맨틱 검색 없음** — 번들 내용 기반 자연어 검색 (예: "지난 주 주차 등록 건수")
- [ ] **AI 신뢰도 표시 없음** — 해석 결과의 확신 정도, 근거 데이터 미표시
- [ ] **지식 충돌 해소 UI 없음** — 동일 항목에 다른 값이 충돌할 때 운영자 판단 인터페이스 없음
- [ ] **데이터 계보(Lineage) 없음** — 번들 변경 이력 (누가, 언제, 무엇을 바꿨는지)

## 3. 객체 수명 주기 관리 ✅ 85%

논문 정의: 생성(Created) → 진행(Processing) → 완료(Done) → 아카이브(Archived) 전 과정 기록

### ✅ 구현된 부분
- **완전한 상태 매트릭스** 구현:
  `pending → cooking → ready → served → paid → archived → canceled → finished → seated → called`
- **각 단계별 컴포넌트 연결**:
  - 생성: `MobileOrderV2` / `ConversationalUI`
  - 조리: `KitchenDisplay` (완료 버튼 → ready)
  - 정산: `CounterPad` + `PaymentModal` (→ paid)
  - 아카이브: `handleCloseSession()` → `/api/session/close`
- 아카이브된 번들은 DB에 보존 → `AdminDashboard`에서 매출 집계에 활용

### ⚠️ 부분 구현
- **삭제 vs 아카이브 혼재**: 호출(`table_calls`), 주차(`table_parkings`), 대기(`table_waitings`) 완료 시 DB에서 **DELETE** 처리 중 → 논문의 아카이빙 원칙과 불일치
- 번들은 아카이브되지만, 별도 운영 테이블 데이터는 삭제

### ❌ 미구현 / 개선 필요
- [ ] **아카이브 데이터 조회 UI 없음** — 날짜별, 타입별 과거 번들 탐색 화면 부재
- [ ] **호출/주차/대기 이력 보존 필요** — 현재 완료 시 DELETE → status='archived'로 UPDATE로 변경 권고
- [ ] **소프트 삭제 복구 불가** — 아카이브 후 복원 메커니즘 없음
- [ ] **수명 주기 이벤트 로그 없음** — 누가, 언제, 어떤 이유로 상태를 변경했는지 기록 없음
- [ ] **보존 정책 설정 없음** — 자동 아카이브 기간, 데이터 내보내기 기능 없음

## 4. 대화형 UX & 마이크로 UI ✅ 75%

논문 정의: 카카오톡 방식의 대화형 인터페이스 + 분절된 마이크로 UI로 모바일 인지 부하 최소화

### ✅ 구현된 부분
- **`ConversationalUI.tsx`** — 완전한 채팅 인터페이스:
  - 텍스트 + 음성 입력 (SpeechRecognition API)
  - TTS 응답 (SpeechSynthesis API)
  - 세션 승인 → 주문 → 결제까지 단계별 대화 흐름
- **MobileOrderV2** — AI 모드 / 메뉴 모드 토글 (고객 QR 접속 → 자동 AI 모드)
- **13개 마이크로 UI 구현**: KitchenDisplay, DisplayBoard, CounterPad, CallManager, WaitingManager, ReservationManager, ParkingManager, PointsManager, HRManager, QRManager, StoreManualEditor, PaymentModal, ReceiptModal
- **실시간 디버그 패널** (`DebugPanel.tsx`) — 색상 구분 로그, 타임스탬프, 접기/펼치기
- 음성 단축 명령어 ("주문", "카운터" 등 → 탭 이동)

### ⚠️ 부분 구현
- 대화형 UI가 **주문/결제 도메인에 한정** — HR, 재고, 예약 등 다른 도메인 대화 불가
- 마이크로 UI들이 **독립 실행 가능하나 상호 연계 흐름** 없음 (예: 대기 → 입장 → 주문 자동 연결)

### ❌ 미구현 / 개선 필요
- [ ] **대화 이력 영속성 없음** — 세션 종료 시 채팅 내역 소멸, 재접속 시 처음부터 시작
- [ ] **멀티 턴 컨텍스트 없음** — 각 메시지가 독립 쿼리, 이전 대화 참조 불가
- [ ] **슬래시 명령어 없음** — `/호출`, `/주차`, `/예약` 등 빠른 실행 명령 미지원
- [ ] **도메인 간 대화 전환 불가** — 주문 중 HR 문의 불가, 컨텍스트 전환 메커니즘 없음
- [ ] **채팅 내 리치 미디어 없음** — 이미지, 버튼 카드, 캐러셀 형태 UI 미지원
- [ ] **마이크로 UI 레지스트리 없음** — 어떤 마이크로 앱이 있는지 운영자가 탐색할 수 있는 카탈로그 부재
